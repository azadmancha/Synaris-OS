## Architecture Overview

### Database Stack

| Component | Technology | Purpose |
|---|---|---|
| Primary Database | PostgreSQL 16 | All relational data |
| Vector Database | Qdrant | Embeddings, semantic search |
| Cache Layer | Redis 7 | Session cache, rate limiting, pub/sub |
| Migrations | Alembic | Schema versioning |
| ORM | SQLAlchemy 2.0 | Async Python ORM |

### Relationships

```
PostgreSQL (Relational)
    │
    ├── Users & Auth (Supabase managed)
    ├── Sessions & Messages
    ├── Learning Profiles
    ├── Knowledge Graph (nodes + edges)
    ├── Study Plans
    ├── Analytics Events
    └── Portfolio Items
            │
            ▼
Qdrant (Vector Store)
    │
    ├── Concept Embeddings
    ├── Session Embeddings
    └── Knowledge Graph Embeddings
            │
            ▼
Redis (Cache)
    │
    ├── Active Sessions
    ├── Rate Limit Counters
    ├── Knowledge Graph Cache
    └── Auth Tokens (if not Supabase-managed)
```

## Core Schema

### Users & Authentication

Supabase Auth manages authentication. We reference Supabase UUIDs.

```sql
-- Managed by Supabase Auth
-- auth.users (id, email, created_at, last_sign_in_at)

-- Extended user profile (Synaris-specific)
CREATE TABLE user_profiles (
    id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
    display_name VARCHAR(100),
    avatar_url TEXT,
    bio TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    -- Preferences
    theme_preference VARCHAR(10) NOT NULL DEFAULT 'system'
        CHECK (theme_preference IN ('light', 'dark', 'system')),
    default_mode VARCHAR(20) NOT NULL DEFAULT 'balanced'
        CHECK (default_mode IN ('quick', 'balanced', 'deep_dive', 'expert')),
    font_size VARCHAR(10) NOT NULL DEFAULT 'medium'
        CHECK (font_size IN ('small', 'medium', 'large')),
    language VARCHAR(10) NOT NULL DEFAULT 'en',
    
    -- Onboarding state
    onboarding_completed BOOLEAN NOT NULL DEFAULT FALSE,
    onboarding_step INTEGER NOT NULL DEFAULT 0
);

-- API keys for external LLM providers
CREATE TABLE user_api_keys (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES user_profiles(id) ON DELETE CASCADE,
    provider VARCHAR(50) NOT NULL,
    encrypted_key BYTEA NOT NULL,
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_used_at TIMESTAMPTZ,
    UNIQUE(user_id, provider)
```

### Messages

```sql
CREATE TABLE messages (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id UUID NOT NULL REFERENCES learning_sessions(id) ON DELETE CASCADE,
    
    -- Content
    role VARCHAR(10) NOT NULL
        CHECK (role IN ('user', 'assistant', 'system')),
    content TEXT NOT NULL,
    content_type VARCHAR(20) NOT NULL DEFAULT 'text'
        CHECK (content_type IN ('text', 'math', 'code', 'diagram', 'image', 'mixed')),

    structured_content JSONB, -- Array of segments
    
    -- AI metadata (for assistant messages)
    model_used VARCHAR(100),
    tokens_used INTEGER,
    processing_time_ms INTEGER,
    
    -- Evaluation (for assistant messages)
    evaluation_score DECIMAL(3,2),
    evaluation JSONB,
    
    -- Context
    parent_message_id UUID REFERENCES messages(id),
    topic VARCHAR(100),
    concepts_mentioned TEXT[],
    
    -- Ordering
    sequence_number INTEGER NOT NULL,
    
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_messages_session_id ON messages(session_id);
CREATE INDEX idx_messages_sequence ON messages(session_id, sequence_number);
CREATE INDEX idx_messages_created_at ON messages(created_at);
CREATE INDEX idx_messages_concepts ON messages USING gin(concepts_mentioned);
```

### Learning Profile (User Mastery)

```sql
CREATE TABLE learning_profiles (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES user_profiles(id) ON DELETE CASCADE,
    
    -- Learning preferences (may be auto-detected)
    preferred_learning_style VARCHAR(20) DEFAULT 'mixed'
        CHECK (preferred_learning_style IN ('visual', 'textual', 'interactive', 'mixed')),
    preferred_depth VARCHAR(10) DEFAULT 'balanced'
        CHECK (preferred_depth IN ('quick', 'balanced', 'deep_dive', 'expert')),
    
    -- Goals
    short_term_goals TEXT[],
    long_term_goals TEXT[],
    target_exams TEXT[],
    
    -- Schedule
    study_minutes_per_day INTEGER,
    preferred_study_times TIME[],
    
    -- Computed metrics
    total_study_hours DECIMAL(10,2) NOT NULL DEFAULT 0,
    average_session_length_minutes DECIMAL(5,2),
    current_streak INTEGER NOT NULL DEFAULT 0,
    longest_streak INTEGER NOT NULL DEFAULT 0,
    last_active_date DATE,
    
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    UNIQUE(user_id)
);

CREATE TABLE concept_mastery (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES user_profiles(id) ON DELETE CASCADE,
    
    -- Concept identification
    concept_name VARCHAR(200) NOT NULL,
    subject VARCHAR(50) NOT NULL,
    topic_path TEXT[], -- Hierarchical topic path
    
    -- Mastery tracking
    mastery_level VARCHAR(20) NOT NULL DEFAULT 'undiscovered'
        CHECK (mastery_level IN (
            'undiscovered', 'introduced', 'developing', 'familiar',
            'mastered', 'can_teach'
        )),
    confidence_score DECIMAL(3,2), -- 0.0 to 1.0
    
    -- Learning history
    times_encountered INTEGER NOT NULL DEFAULT 0,
    times_correct INTEGER, -- For quizzed concepts
    times_incorrect INTEGER,
    last_reviewed_at TIMESTAMPTZ,
    next_review_at TIMESTAMPTZ, -- For spaced repetition
    
    -- Metadata
    source_sessions UUID[], -- Session IDs where this was discussed
    prerequisites TEXT[], -- Required concepts
    related_concepts TEXT[],
    
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    UNIQUE(user_id, concept_name, subject)
);

CREATE INDEX idx_mastery_user ON concept_mastery(user_id);
CREATE INDEX idx_mastery_subject ON concept_mastery(subject);
CREATE INDEX idx_mastery_level ON concept_mastery(mastery_level);
CREATE INDEX idx_mastery_next_review ON concept_mastery(next_review_at)
    WHERE next_review_at IS NOT NULL;
```

## Knowledge Graph

```sql
CREATE TABLE knowledge_graph_nodes (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    
    concept_name VARCHAR(200) NOT NULL,
    subject VARCHAR(50) NOT NULL,
    
    -- Description
    short_description TEXT,
    long_description TEXT,
    
    -- Category
    node_type VARCHAR(30) NOT NULL DEFAULT 'concept'
        CHECK (node_type IN ('concept', 'formula', 'theorem', 'law', 'method',
                             'field', 'application', 'example')),
    
    -- Metadata
    difficulty_level INTEGER CHECK (difficulty_level >= 1 AND difficulty_level <= 5),
    wikipedia_url TEXT,
    wikidata_id VARCHAR(50),
    
    -- Embedding reference
    embedding_id UUID,
    
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    UNIQUE(concept_name, subject)
);

CREATE TABLE knowledge_graph_edges (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_node_id UUID NOT NULL REFERENCES knowledge_graph_nodes(id) ON DELETE CASCADE,
    target_node_id UUID NOT NULL REFERENCES knowledge_graph_nodes(id) ON DELETE CASCADE,
    
    relationship_type VARCHAR(30) NOT NULL
        CHECK (relationship_type IN (
            'prerequisite', 'builds_on', 'related_to', 'generalization',
            'specialization', 'analogous_to', 'application_of',
            'contradicts', 'proven_by', 'example_of'
        )),
    
    weight DECIMAL(3,2) DEFAULT 1.0, -- Strength of relationship
    description TEXT,
    
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    UNIQUE(source_node_id, target_node_id, relationship_type),
    CHECK (source_node_id != target_node_id)
);

CREATE INDEX idx_kg_edges_source ON knowledge_graph_edges(source_node_id);
CREATE INDEX idx_kg_edges_target ON knowledge_graph_edges(target_node_id);
CREATE INDEX idx_kg_edges_type ON knowledge_graph_edges(relationship_type);
```

## Vector Store Integration (Qdrant)

### Collections

| Collection Name | Content | Vector Size | Purpose |
|---|---|---|---|
| `concepts` | Concept embeddings | 1024 | Semantic concept search |
| `sessions` | Session summary embeddings | 1024 | "Continue where you left off" |
| `messages` | Message content embeddings | 1024 | Semantic message search |
| `knowledge_graph` | Node embeddings | 1024 | Graph-based similarity search |
| `user_knowledge` | Per-user concept state | 1024 | Personalized retrieval |

### Qdrant Point Structure

```json
{
  "id": "uuid",
  "vector": [0.1, 0.2, ...],
  "payload": {
    "type": "concept",
    "concept_name": "entropy",
    "subject": "physics",
    "mastery_level": "developing",
    "user_id": "uuid",
    "created_at": "ISO8601"
  }
}
```

### Retrieval Strategy

```python
# When looking for related concepts:
# 1. Encode the query with BGE-M3 or Jina Embeddings
# 2. Search Qdrant for top-k similar vectors
# 3. Filter by subject if known
# 4. Boost by graph proximity (if knowledge graph matches)
# 5. Return ranked results with relevance scores
```

## Redis Cache Layer

### Cache Keys

| Key Pattern | TTL | Content |
|---|---|---|
| `session:{id}` | 1 hour | Active session context |
| `session:{id}:messages` | 30 min | Recent messages for context |
| `user:{id}:profile` | 5 min | Cached learning profile |
| `kg:related:{concept}` | 1 hour | Related concepts from KG |
| `rate_limit:{user_id}:{endpoint}` | Dynamic | Rate limit counters |
| `auth:{token}` | Token expiry | Auth session cache |

| Channel | Message | Use Case |
|---|---|---|
| `session:update` | Session ID + update type | Real-time dashboard updates |
| `learning:event` | Learning event payload | Analytics pipeline |
| `notifications:user:{id}` | Notification payload | In-app notifications |

## Performance & Indexing

### Critical Indexes

```sql
-- Sessions
CREATE INDEX CONCURRENTLY idx_sessions_active
    ON learning_sessions(user_id, status)
    WHERE status = 'active';

-- Messages
CREATE INDEX CONCURRENTLY idx_messages_recent
    ON messages(session_id, sequence_number DESC);

-- Concept mastery - for spaced repetition queries
CREATE INDEX CONCURRENTLY idx_mastery_review_due
    ON concept_mastery(user_id, next_review_at)
    WHERE next_review_at IS NOT NULL AND next_review_at <= NOW();

-- Study plans
CREATE INDEX CONCURRENTLY idx_tasks_today
    ON study_plan_tasks(plan_id, scheduled_date)
    WHERE completed_at IS NULL;

-- Analytics time-series queries
CREATE INDEX CONCURRENTLY idx_analytics_daily
    ON analytics_events(created_at)
    WHERE event_type = 'session_completed';
```

### Connection Pooling

```python
# SQLAlchemy async engine
from sqlalchemy.ext.asyncio import create_async_engine

engine = create_async_engine(
    "postgresql+asyncpg://user:pass@host:5432/synaris",
    pool_size=20,
    max_overflow=10,
    pool_pre_ping=True,
    pool_recycle=3600,  # Recycle connections after 1 hour
    echo=False
)
```

### Query Optimization Guidelines

1. **Use connection pooling** — Minimize connection overhead
2. **Prefetch with selectinload** — Avoid N+1 queries
3. **Batch inserts** — Use `bulk_insert_mappings` for batch operations
4. **Partial indexes** — Index only relevant rows
5. **Covering indexes** — Include all queried columns
6. **Materialized views** — For complex dashboard queries

## Backup & Recovery

### Backup Strategy

| Backup Type | Frequency | Retention | Storage |
|---|---|---|---|
| Full database | Daily | 30 days | S3/Cloudflare R2 |
| WAL archives | Continuous | 7 days | S3/Cloudflare R2 |
| Qdrant snapshot | Daily | 7 days | S3/Cloudflare R2 |
| Redis dump | Every 6 hours | 24 hours | Local + S3 |

### Recovery Plan

1. **Point-in-time recovery** — Restore PostgreSQL from WAL
2. **Full restore** — From latest full backup + WAL
3. **Qdrant restore** — From latest snapshot
4. **Warm standby** — Read replica for failover

### Disaster Recovery

```yaml
# docker-compose disaster recovery
recovery:
  steps:
    1. Spin up new PostgreSQL instance
    2. Restore from latest backup
    3. Apply WAL to desired point-in-time
    4. Restore Qdrant from snapshot
    5. Verify data integrity
    6. Point application to new DB
    7. Verify application health
    8. Decommission old DB
```
